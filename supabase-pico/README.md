# Supabase Pico (512 MiB, headless)

[![Deploy to Miget](https://miget.com/deploy-to-miget.svg)](https://miget.com/deploy?repo=https://github.com/deployable-sh/stacks&path=supabase-pico&type=stack)

Supabase's API surface - Postgres, Auth (GoTrue) and REST (PostgREST) -
in **512 MiB total**. The trick that closes the math: Kong is replaced by
a ~6 MiB nginx gateway that does exactly what Kong did for these routes
(path routing, apikey allowlist, Authorization fallback, CORS, and the
same JWT_SECRET -> API-key derivation at start). No dashboard on the
server; run [Studio from your laptop](#dashboard) when you need it.

Want Studio on the server? That's [supabase-nano](../supabase-nano)
(1 GiB). Comfortable margins? [supabase-lite](../supabase-lite) (2 GiB).

## The sizing

Measured under load (signups + RLS queries via supabase-js, 600 concurrent
REST requests, pgbench from an external client) against hard no-swap
memory limits:

| Service | RAM | Peak seen | What made it fit |
|---|---|---|---|
| `gateway` | 16 | 3 | it's nginx with one worker |
| `auth` | 48 | 33 | nothing - GoTrue is just small |
| `rest` | 224 | fills its allowance | `PGRST_DB_POOL=2` (GHC expands to whatever it gets; never OOMs) |
| `db` | 224 | ~120 | `shared_buffers=64MB`, 30 connections, pinned autovacuum/WAL |

## What the gateway does NOT do

Compared to the Kong stacks: no Studio routes, no `sb_*` opaque
publishable/secret key translation (legacy `ANON_KEY`/`SERVICE_ROLE_KEY`
JWTs only), no Realtime/Storage/Functions routes (those services aren't
here). `/auth/v1/*`, `/rest/v1/*` and `/graphql/v1` behave identically,
including open access to `/auth/v1/verify|callback|authorize` for email
links and OAuth.

## Trade-offs (read before production)

- **2 PostgREST pool connections** - hobby APIs yes, hot paths no. Raise
  `PGRST_DB_POOL` only alongside `rest` RAM (pool 10 costs ~290 MiB).
- **30 Postgres connections** total, ~25 usable by your tooling.
- Bulk imports are the memory danger zone: dirty pages from a huge
  single COPY outrun writeback and trip the OOM killer. Measured
  boundary: repeated ~200k-row batches ran clean at 6.6k tps; ~500k-row
  batches killed the database. `max_wal_size=256MB` and a single
  autovacuum worker are pinned to soften this. For a big one-off import,
  chunk it or temporarily raise `db` RAM.
- A schema with dozens of tables grows PostgREST's cache - move up to
  nano/lite (same database image and volume layout).

## Topology

| Service | Role | Public |
|---|---|---|
| `gateway` | nginx: routing + apikey + CORS (`:5000`) | yes |
| `auth` | GoTrue - signups, logins, JWTs | via gateway |
| `rest` | PostgREST auto API | via gateway |
| `db` | supabase-postgres (own image, own volume) | no |

## Local

```bash
cp .env.upstream-example .env    # fill in the secrets
docker compose up -d --build
curl -s http://localhost:5000/health
```

## Dashboard

Studio is stateless - everything lives in Postgres - so it runs anywhere,
including your laptop, via [`studio.local.yaml`](studio.local.yaml):

```bash
# against a local pico on the same machine:
docker compose -f compose.yaml -f studio.local.yaml up -d studio meta
open http://localhost:3000
```

Against a deployed instance, point `SUPABASE_PUBLIC_URL` at the gateway
domain and `STUDIO_POSTGRES_HOST` at a reachable Postgres (expose the
`db` app's TCP port 5432 on Miget - use a strong password). The file
header shows the full invocation.

## On Miget

Create a Compose Stack pointing at this repo, path `supabase-pico`.
Required variable: `POSTGRES_PASSWORD`. `JWT_SECRET` is generated by the
platform and the gateway derives `ANON_KEY`/`SERVICE_ROLE_KEY` from it at
start (same derivation as the Kong stacks) - or set all three yourself
with supabase's
[key generator](https://supabase.com/docs/guides/self-hosting/docker#securing-your-services).
`SITE_URL`/`API_EXTERNAL_URL` are wired from the gateway app's public
domain.
